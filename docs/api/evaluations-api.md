 

# Tekka 評価API仕様書

## 1. 概要

本仕様書では、TekkaのPhase 1で使用する評価APIを定義する。

評価APIでは、ログインユーザーが記事へ高評価または低評価を付ける。すでに付けている評価と同じ評価を送信した場合は評価を取り消し、異なる評価を送信した場合は評価を切り替える。

データベース操作には、LaravelのEloquent ORMを使用する。

評価の作成・更新・削除、記事スコアの再計算、および投稿者のユーザースコアの再計算は、同一のDBトランザクション内で行う。

API全体の共通ルールについては、`api-common.md`を参照する。

# 3. 記事評価

# 4. 評価のトグル動作

1人のユーザーは、1つの記事につき1件だけ評価を持つ。

送信された`value`と現在の評価状態に応じて、次の処理を行う。

| 現在の状態 | 送信値 | 処理後の状態 | DB操作     |
| ---------- | ------ | ------------ | ---------- |
| 未評価     | `high` | `high`       | 評価を作成 |
| 未評価     | `low`  | `low`        | 評価を作成 |
| `high`     | `high` | 未評価       | 評価を削除 |
| `low`      | `low`  | 未評価       | 評価を削除 |
| `high`     | `low`  | `low`        | 評価を更新 |
| `low`      | `high` | `high`       | 評価を更新 |

処理の判定は次のとおりとする。

```text
評価レコードが存在しない
→ 送信されたvalueで評価を作成する

現在のvalueと送信されたvalueが同じ
→ 評価レコードを削除する

現在のvalueと送信されたvalueが異なる
→ 送信されたvalueへ評価を更新する
```

未評価状態は、`article_evaluations`テーブルに該当する評価レコードが存在しない状態で表す。

# 5. Eloquent ORMによる処理方針

評価レコードは、ログインユーザーと対象記事の組み合わせで取得する。

```text
article_id = 対象記事のID
user_id = ログインユーザーのID
```

生のSQL文字列を組み立てず、Eloquentモデル、リレーション、およびクエリビルダを使用する。

主な操作方針は次のとおりとする。

| 処理               | Eloquentでの方針                                   |
| ------------------ | -------------------------------------------------- |
| 記事取得           | `Article`モデルから取得                            |
| 現在の評価取得     | 記事またはユーザーの評価リレーションから取得       |
| 評価作成           | モデルまたはリレーションの`create()`を使用         |
| 評価更新           | モデルの値を変更して`save()`または`update()`を使用 |
| 評価削除           | モデルの`delete()`を使用                           |
| 評価件数集計       | `where()`と`count()`を使用                         |
| 記事スコア保存     | `Article`モデルへ保存                              |
| ユーザースコア保存 | `User`モデルへ保存                                 |

トランザクションにはLaravelの`DB::transaction()`を使用する。

`DB::transaction()`はトランザクション境界の管理に使用し、評価、記事、ユーザーの操作自体はEloquent ORMで行う。

# 6. 評価できない記事

## 6.1 自分の記事

投稿者本人は、自分の記事を評価できない。

```text
ログインユーザーID
=
記事のuser_id

→ 403 Forbidden
```

フロントエンド側でも評価ボタンを非表示または無効化できるが、最終的な認可はLaravel側で行う。

## 6.2 論理削除済みの記事

論理削除済みの記事には評価操作を行えない。

次の操作はすべて不可とする。

- 新しい評価の作成
- 高評価と低評価の切り替え
- 既存評価の取り消し

論理削除済みの記事は、存在しない記事と同様に`404 Not Found`として扱う。

## 6.3 一覧非表示の記事

記事スコアまたは投稿者のユーザースコアによって一覧から非表示となっている記事でも、論理削除されていなければ評価できる。

| 記事の状態                       | 評価 |
| -------------------------------- | ---: |
| 通常の記事                       | 可能 |
| 記事スコアが`-3`以下             | 可能 |
| 投稿者のユーザースコアが`90`以下 | 可能 |
| 論理削除済み                     | 不可 |

これは、一覧から非表示となっている記事でも、記事詳細URLから直接閲覧できる仕様と一致させるためである。

# 7. 記事スコアの再計算

評価操作後、対象記事の記事スコアを再計算する。

```text
記事スコア
= highの評価数 - lowの評価数
```

例:

```text
high: 5件
low: 2件

記事スコア
= 5 - 2
= 3
```

## 7.1 再集計方式

現在のスコアへ単純に加算・減算する方式は使用しない。

```text
highが追加された
→ articles.scoreへ1を加算する
```

上記のような差分更新ではなく、評価レコードをEloquentで再集計する。

```text
highの評価件数を取得する
lowの評価件数を取得する
↓
high件数 - low件数
↓
articles.scoreへ保存する
```

再集計方式を使用することで、`articles.score`と実際の評価レコードがずれにくくする。

# 8. ユーザースコアの再計算

記事スコアの更新後、記事投稿者のユーザースコアを再計算する。

ユーザースコアの計算対象は、投稿者が所有する論理削除されていない記事とする。

## 8.1 記事ごとの影響値

1つの記事がユーザースコアへ与える影響値は、記事スコアを`-3`から`3`の範囲へ制限した値とする。

|  記事スコア | ユーザースコアへの影響値 |
| ----------: | -----------------------: |
|     `4`以上 |                      `3` |
| `-3`から`3` |       記事スコアと同じ値 |
|    `-4`以下 |                     `-3` |

例:

```text
記事スコアが5
→ 影響値は3

記事スコアが2
→ 影響値は2

記事スコアが-4
→ 影響値は-3
```

## 8.2 計算式

ユーザースコアは次の式で計算する。

```text
ユーザースコア
= 100 + 論理削除されていない各記事の影響値の合計
```

例:

```text
記事A: score 5
→ 影響値 3

記事B: score 2
→ 影響値 2

記事C: score -4
→ 影響値 -3

ユーザースコア
= 100 + 3 + 2 - 3
= 102
```

ユーザースコアの再集計には、投稿者の記事リレーションおよびEloquentのコレクションまたはクエリビルダを使用する。

論理削除済みの記事は計算対象に含めない。

# 9. トランザクション

評価操作、記事スコア更新、およびユーザースコア更新は、同一のDBトランザクション内で実行する。

```text
1. 論理削除されていない記事を取得する
2. ログインユーザーが投稿者本人ではないことを確認する
3. ログインユーザーの現在の評価を取得する
4. 評価を作成・更新・削除する
5. 対象記事のhigh評価数を再集計する
6. 対象記事のlow評価数を再集計する
7. 記事スコアを更新する
8. 投稿者のユーザースコアを再計算する
9. users.scoreを更新する
10. すべて成功した場合のみ確定する
```

途中で処理に失敗した場合は、次の変更をすべて取り消す。

- 評価の作成
- 評価の更新
- 評価の削除
- 記事スコアの変更
- ユーザースコアの変更

## 9.1 評価の一意性

DBの複合ユニーク制約により、同一ユーザーが同一記事に複数の評価を持つことを防ぐ。

```text
UNIQUE (user_id, article_id)
```

## 9.2 重複リクエストへの対応

Phase 1では、高度な排他制御や複雑なロック処理は実装しない。

フロントエンドでは、評価リクエストの処理中に評価ボタンを一時的に無効化し、短時間の連打による重複リクエストを抑制する。

DBの複合ユニーク制約は、評価レコードの重複作成を防ぐ最終的な制約として機能する。

# 10. 成功レスポンス

## 10.1 レスポンス項目

評価操作後、記事詳細画面で表示を更新するため、成功レスポンスには次の項目を含める。

| 項目                      | 型            | `null` | 内容                                                   |
| ------------------------- | ------------- | -----: | ------------------------------------------------------ |
| `article_id`              | integer       |   不可 | 評価対象の記事ID                                       |
| `current_user_evaluation` | string / null |     可 | 評価操作後にログインユーザーが対象記事へ付けている評価 |
| `article_score`           | integer       |   不可 | 評価操作後の記事スコア                                 |

`current_user_evaluation`は、ユーザースコアを表す項目ではない。

値は次のいずれかとする。

| 値       | 意味                           |
| -------- | ------------------------------ |
| `"high"` | 評価操作後、高評価が付いている |
| `"low"`  | 評価操作後、低評価が付いている |
| `null`   | 評価操作後、未評価となっている |

投稿者のユーザースコアはLaravel側で再計算して保存するが、記事詳細画面では直接使用しないため、評価APIのレスポンスには含めない。

## 10.2 評価を作成した場合

```text

```

# 11. フロントエンドでの利用

React側は、評価APIの成功レスポンスを使用して記事詳細画面の表示を更新する。

```text
current_user_evaluation
→ 高評価・低評価ボタンの選択状態を更新する

article_score
→ 画面に表示している記事スコアを更新する
```

例:

```js
const response = await axios.post(
  `/api/articles/${articleId}/evaluation`,
  {
    value: "high",
  }
);

setCurrentUserEvaluation(
  response.data.data.current_user_evaluation
);

setArticleScore(
  response.data.data.article_score
);
```

評価操作後に記事詳細APIを再取得しなくても、評価ボタンの状態と記事スコアを更新できる。

# 15. バリデーションエラー

## 15.1 エラー条件

次の場合は、`422 Unprocessable Content`を返す。

- `value`が指定されていない
- `value`が文字列ではない
- `value`が`high`でも`low`でもない

# 16. 処理内容

```text
1. ログインユーザーを取得する
2. valueを検証する
3. 論理削除されていない対象記事を取得する
4. ログインユーザーが投稿者本人ではないことを確認する
5. DBトランザクションを開始する
6. ログインユーザーの現在の評価を取得する
7. 現在の評価状態に応じて評価を作成・更新・削除する
8. 対象記事のhigh評価数を再集計する
9. 対象記事のlow評価数を再集計する
10. 記事スコアを更新する
11. 投稿者のユーザースコアを再計算する
12. users.scoreを更新する
13. トランザクションを確定する
14. 評価操作後の状態と記事スコアを返す
```
