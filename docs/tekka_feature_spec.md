# Tekka 機能ベース仕様書

## 1. 概要

Tekka は、Qiita や Zenn のような技術記事投稿サイトを参考にした、技術記事の作成・閲覧・評価を行う Web アプリケーションである。

本仕様書では、現時点で決定済みの機能範囲と技術構成を整理する。DB のテーブル設計、カラム設計、リレーション設計は本仕様書では扱わない。API の詳細な URL 設計、レスポンス形式、エラー形式、ステータスコード、画面ごとの詳細レイアウトは、後続の API 仕様書または画面設計で扱う。

## 2. 目的

Tekka の目的は、ユーザーが技術記事を投稿し、他のユーザーの記事を閲覧・検索・評価できる場を作ることである。

主な目的は次の通りである。

- 技術記事を投稿できる
- 技術記事を一覧・詳細で閲覧できる
- Markdown 形式で記事を書ける
- タグやキーワードで記事を探せる
- 記事に対して高評価・低評価を付けられる
- 記事評価をもとに記事やユーザーの表示制御を行う

## 3. 使用技術

### 3.1 バックエンド

- 言語: PHP
- フレームワーク: Laravel
- 役割: API サーバー

Laravel は画面生成を担当せず、React フロントエンドに対して API を提供する。

主な担当範囲は次の通りである。

- 認証処理
- ログイン状態の管理
- 記事データの取得・作成・更新・削除
- タグ情報の取得・管理
- キーワード検索処理
- 記事評価処理
- スコア計算処理
- ユーザー情報の取得
- バリデーション
- 権限チェック
- JSON レスポンスの返却

### 3.2 フロントエンド

- ライブラリ: React
- 構成: SPA
- ルーティング: React Router
- API 通信: axios
- Markdown表示: react-markdown
- CSS: 素の CSS
- Inertia: 使用しない
- Tailwind CSS: 使用しない

React は画面表示とユーザー操作を担当する。

主な担当範囲は次の通りである。

- ページルーティング
- 画面表示
- フォーム入力
- API 通信
- axios の共通設定
- ログイン状態に応じた表示切り替え
- Markdown 入力・表示に関する UI
- タグ入力・タグ表示に関する UI
- 高評価・低評価ボタンの表示と操作
- 検索フォームの表示と操作

API のベース URL や認証関連の設定は、axios の共通設定で管理する。

### 3.3 データベース

- DB: MySQL

DB のテーブル設計、カラム設計、リレーション設計は本仕様書では扱わない。

### 3.4 認証

- 認証方式: Laravel Sanctum の SPA 認証
- 認証状態の管理: Cookie ベース
- CSRF 保護: 有効
- JWT 認証: 採用しない

Laravel Sanctum を用いて、React SPA から Laravel API への認証付きリクエストを扱う。認証関連の API 通信設定は、フロントエンド側の共通設定として管理する。

### 3.5 開発環境

- ローカル開発環境: ServBay

React と Laravel は同一リポジトリで管理する。

## 4. システム構成

Tekka は、React SPA と Laravel API による分離構成とする。

```text
Browser
  ↓
React SPA
  ↓ HTTP API
Laravel API
  ↓
MySQL
```

Laravel は API サーバーとして JSON を返す。React は API から取得したデータをもとに画面を描画する。

## 6. 機能仕様

### 6.1 ユーザー登録

未登録ユーザーは、アカウントを作成できる。

登録する内容は次の通りである。

- ユーザー名
- メールアドレス
- パスワード

ユーザー名は一意とする。メールアドレスは公開しない。

ユーザー名の入力制限は次の通りである。

- 3文字以上30文字以内

登録後に自動ログインするかどうかは、後続の画面設計または認証フロー設計で決定する。

### 6.2 ログイン

登録済みユーザーは、メールアドレスとパスワードを用いてログインできる。

ログイン後に可能になる操作は次の通りである。

- 記事投稿
- 自分の記事の編集
- 自分の記事の削除
- 記事への高評価
- 記事への低評価
- プロフィール編集

ログイン失敗時のエラー表示、ログイン後の遷移先は、後続の画面設計で決定する。

### 6.3 ログアウト

ログイン中のユーザーは、ログアウトできる。

ログアウト後は、ログインが必要な操作を実行できない。

### 6.4 プロフィール編集

ログイン中のユーザーは、自分のプロフィールを編集できる。

編集可能な内容は次の通りである。

- ユーザー名
- プロフィール文

ユーザー名は変更可能とする。ユーザーページの URL はユーザー ID ベースであるため、ユーザー名を変更しても URL は変わらない。

プロフィール文の入力制限は次の通りである。

- 500文字以内

アイコン画像は Phase 1 では扱わない。アイコン画像に関する API・画面・アップロード機能も Phase 1 では実装しない。

### 6.5 記事一覧

ユーザーは、投稿された記事の一覧を閲覧できる。

記事一覧では、記事を探しやすくするために、記事の概要情報を表示する。

想定する表示内容は次の通りである。

- 記事タイトル
- 投稿者情報
- 投稿日時
- タグ
- 記事の概要
- 記事スコア

記事一覧は、作成日時の新しい順で表示する。更新日時順の並び替え、スコア順の並び替えは Phase 1 では実装しない。

次の記事は記事一覧に表示しない。

- 論理削除された記事
- 記事スコアが -3 以下の記事
- ユーザースコアが 90 以下のユーザーの記事

ページネーションの有無や表示件数は、後続の画面設計または API 仕様書で決定する。

### 6.6 記事詳細

ユーザーは、記事の詳細を閲覧できる。

記事詳細では、Markdown 原文として保存された記事本文を React 側で表示用の形式に変換して表示する。

想定する表示内容は次の通りである。

- 記事タイトル
- 記事本文
- 投稿者情報
- 投稿日時
- 更新日時
- タグ
- 高評価・低評価の状態
- 記事スコア

記事詳細ページの URL は `/articles/{id}` とする。記事タイトルは URL 識別子には使わない。slug は Phase 1 では実装しない。

低評価により一覧非表示となった記事でも、記事詳細ページへの直接アクセスは許可する。ただし、論理削除された記事の詳細ページへ直接アクセスした場合は 404 とする。

### 6.7 記事投稿

ログイン中のユーザーは、新しい記事を投稿できる。

記事投稿では、Markdown 形式で本文を入力できる。

入力内容は次の通りである。

- 記事タイトル
- 記事本文
- タグ

記事は投稿時点で公開状態とする。下書き機能と非公開記事機能は Phase 1 では実装しない。

入力制限は次の通りである。

- 記事タイトル: 1文字以上100文字以内
- 記事本文: 1文字以上50,000文字以内
- タグ: 1記事あたり最大5個

Phase 1 では、投稿画面にリアルタイム Markdown プレビューは実装しない。投稿後の記事詳細表示では Markdown 表示に対応する。リアルタイム Markdown プレビューは、余裕があれば後の段階で実装する。

### 6.8 記事編集

ログイン中のユーザーは、自分が投稿した記事を編集できる。

編集可能な内容は次の通りである。

- 記事タイトル
- 記事本文
- タグ

他人の記事は編集できない。記事編集ページの URL は `/articles/{id}/edit` とする。

編集履歴は保持しない。

### 6.9 記事削除

ログイン中のユーザーは、自分が投稿した記事を削除できる。

他人の記事は削除できない。

記事削除は論理削除とする。論理削除された記事は、記事一覧・検索結果・タグ別記事一覧・ユーザーページには表示しない。投稿者本人であっても、削除済み記事は通常画面から閲覧できない。

復元機能は Phase 1 では実装しない。

### 6.10 Markdown 対応

記事本文は Markdown に対応する。

ユーザーは Markdown 記法で記事を書き、記事詳細画面では読みやすい形式で表示される。

Markdown の仕様は次の通りである。

- 記事本文は Markdown 形式で入力する
- 記事本文は Markdown 原文として保存する
- Markdown の表示変換には React 側で react-markdown を使用する
- 使用できる Markdown 記法は、標準的な Markdown ライブラリで扱える範囲とする
- 独自 Markdown 記法は実装しない
- Markdown 内の HTML は許可しない。react-markdown の `rehypeRaw` は使用しない
- XSS 対策として、`ReactMarkdown` の `skipHtml` プロパティを `true` に設定する
- react-markdown が提供する安全な既定の URL 変換処理を使用し、危険な独自 `urlTransform` は設定しない
- コードブロックは使用可能とする

```tsx
<ReactMarkdown skipHtml>{body}</ReactMarkdown>
```

`skipHtml`はboolean型のプロパティであり、`true`を指定するとMarkdown本文に含まれるHTMLを無視する。`body`はMarkdown原文を表すstring型の値である。

HTML を許可すると、許可タグ・サニタイズ・XSS 対策の範囲を決める必要があるため、Phase 1 では許可しない。

### 6.11 タグ

記事にはタグを付けられる。

タグは、記事の分類やタグ別記事一覧に利用する。

タグ仕様は次の通りである。

- タグは自由入力とする
- 既存タグは再利用可能とする
- 存在しないタグは、記事投稿・編集時に自動作成する
- 1つの記事につけられるタグは最大5個までとする
- タグ名は一意とする
- タグ名の大文字・小文字は区別しない
- タグ名は前後の空白を除去して扱う
- 同じ記事に同じタグを重複して付けることはできない
- タグ名は1文字以上30文字以内とする

次のタグ名は同じタグとして扱う。

```text
Laravel
laravel
LARAVEL
```

タグ別記事一覧は、作成日時の新しい順で表示する。

### 6.12 キーワード検索

ユーザーは、キーワードを用いて記事を検索できる。

検索仕様は次の通りである。

- キーワード検索の対象は、記事タイトルと記事本文のみとする
- タグはキーワード検索対象には含めない
- タグによる絞り込みは、キーワード検索とは別にタグ別記事一覧として扱う
- 検索方法は部分一致とする
- 検索結果は作成日時の新しい順で表示する

想定する URL は次の通りである。

```text
/search?q=laravel
/tags?tag=laravel
```

次の記事は検索結果に表示しない。

- 論理削除された記事
- 記事スコアが -3 以下の記事
- ユーザースコアが 90 以下のユーザーの記事

### 6.13 ユーザーページ

ユーザーごとのページを表示する。

ユーザーページの URL は `/users/{id}` とする。ユーザー名は URL 識別子には使わない。

表示内容は次の通りである。

- ユーザー名
- プロフィール文
- ユーザースコア
- そのユーザーの投稿記事一覧

ユーザーページでは、記事スコアやユーザースコアによる非表示判定を行わない。ただし、論理削除された記事は表示しない。

「ユーザーページでは全ての記事を表示する」という場合の「全て」は、削除されていない記事を指す。

### 6.14 記事への評価

ログイン中のユーザーは、記事に高評価または低評価を付けられる。未ログインユーザーは評価できない。

評価仕様は次の通りである。

- 評価値は `high` または `low` とする
- 1ユーザーは1記事につき1つの評価のみ持てる
- 同じ評価をもう一度押した場合は、評価を取り消す
- 別の評価を押した場合は、評価を切り替える
- 投稿者本人は自分の記事を評価できない
- 低評価理由の入力は実装しない

評価の挙動例は次の通りである。

```text
未評価 -> 高評価を押す -> high
high -> 高評価を押す -> 未評価
high -> 低評価を押す -> low
low -> 低評価を押す -> 未評価
low -> 高評価を押す -> high
```

### 6.15 記事スコア算出

記事ごとにスコアを算出する。

記事スコアは、高評価数から低評価数を引いた値とする。

```text
記事スコア = 高評価数 - 低評価数
```

例は次の通りである。

```text
高評価 5、低評価 2 -> 記事スコア 3
高評価 1、低評価 4 -> 記事スコア -3
```

記事スコアは次の機能に関係する。

- 記事一覧での表示
- 記事詳細での表示
- 記事の非表示判定
- ユーザースコア算出

### 6.16 ユーザースコア算出

ユーザーごとにスコアを算出する。

ユーザースコア仕様は次の通りである。

- ユーザースコアの初期値は100とする
- ユーザースコアは、そのユーザーが投稿した削除されていない記事を対象に算出する
- 削除済み記事はユーザースコアの対象に含めない
- 低評価により一覧非表示となった記事も、ユーザースコアの対象に含める
- 評価数0の記事も、ユーザースコアの対象に含める
- 1つの記事がユーザースコアに与える影響は、最大+3、最小-3とする
- ユーザースコアは、初期値100に各記事の影響値を加算して算出する

計算式は次の通りである。

```text
記事スコア = 高評価数 - 低評価数

記事の影響値 =
記事スコアが 3 より大きい場合: 3
記事スコアが -3 より小さい場合: -3
それ以外の場合: 記事スコア

ユーザースコア = 100 + 各記事の影響値の合計
```

例は次の通りである。

```text
記事A: 記事スコア 5 -> 影響値 +3
記事B: 記事スコア 2 -> 影響値 +2
記事C: 記事スコア -4 -> 影響値 -3

ユーザースコア = 100 + 3 + 2 - 3
ユーザースコア = 102
```

### 6.17 スコア保存方針

スコア保存方針は次の通りである。

- 記事スコアは DB に保存する
- ユーザースコアは DB に保存する
- 評価が作成・変更・削除されたタイミングで、対象記事の記事スコアを再計算して保存する
- 記事スコアが変更された場合、対象記事の投稿者のユーザースコアを再計算して保存する
- 一覧表示では、保存済みの記事スコア・ユーザースコアを参照する
- スコア計算式を後から変更した場合、既存記事・既存ユーザーのスコアを再計算する必要がある

この章は機能上の保存方針を示すものであり、テーブル名・カラム名・型までは本仕様書には書かない。

### 6.18 低評価記事の一覧非表示

記事スコアが -3 以下の記事は、次の画面に表示しない。

- 記事一覧
- 検索結果
- タグ別記事一覧

例は次の通りである。

```text
記事スコア -1 -> 表示
記事スコア -2 -> 表示
記事スコア -3 -> 一覧・検索結果・タグ別一覧では非表示
記事スコア -4 -> 一覧・検索結果・タグ別一覧では非表示
```

低評価により一覧非表示となった記事でも、記事詳細ページへの直接アクセスは許可する。ユーザーページでは、記事スコアによる非表示判定を行わない。

### 6.19 低評価ユーザーの記事の一覧非表示

ユーザースコアが 90 以下のユーザーの記事は、次の画面に表示しない。

- 記事一覧
- 検索結果
- タグ別記事一覧

例は次の通りである。

```text
初期ユーザースコア: 100

-3 の影響を持つ記事が1本 -> 97
-3 の影響を持つ記事が2本 -> 94
-3 の影響を持つ記事が3本 -> 91
-3 の影響を持つ記事が4本 -> 88
```

ユーザースコアが 90 以下で非表示対象になるため、1本の記事だけでユーザー全体が非表示対象になることはない。

ユーザーページでは、ユーザースコアによる非表示判定を行わない。記事詳細ページへの直接アクセスは許可する。

### 6.20 非表示条件のユーザー設定

Phase 1 では、非表示条件の閾値は固定値とする。

Phase 1 では、ユーザーごとに非表示閾値を変更する機能は実装しない。

将来的な拡張として、ログインユーザーごとに非表示条件の閾値を変更できる設定機能を追加する可能性がある。

ユーザー設定で閾値を変更した場合、その設定は設定した本人の記事一覧・検索結果・タグ別記事一覧の表示にのみ適用する。他のユーザーの表示結果には影響しない。

## 7. 権限方針

権限に関する基本方針は次の通りである。

- 未ログインユーザーは記事を閲覧できる
- 未ログインユーザーは記事投稿できない
- 未ログインユーザーは記事を評価できない
- ログインユーザーは記事を投稿できる
- ログインユーザーは自分の記事を編集できる
- ログインユーザーは自分の記事を削除できる
- ログインユーザーは自分のプロフィールを編集できる
- ログインユーザーは他人の記事を編集できない
- ログインユーザーは他人の記事を削除できない
- ログインユーザーは他人のプロフィールを編集できない
- 投稿者本人は自分の記事を評価できない

Phase 1 では管理者機能は実装しない。全ユーザーを通常ユーザーとして扱い、管理者用 API・管理者用画面・管理者権限を前提にした機能は作成しない。

## 8. API 方針

Laravel は React に対して API を提供する。

API の基本方針は次の通りである。

- API URL は `/api/...` で統一する
- 基本的に REST 風の URL 設計にする
- レスポンスは JSON とする
- エラー時も JSON とする
- 認証が必要な操作では Sanctum を用いる
- 入力値の検証は Laravel 側で行う
- 権限チェックは Laravel 側で行う
- React 側のバリデーションは補助的なものとする

API ごとの詳細な URL、リクエスト形式、レスポンス形式、エラー形式、ステータスコードは、後で API 仕様書を作成する段階で決める。

## 9. React 側の画面候補

React 側で作成する画面候補は次の通りである。

- トップページ
- 記事一覧ページ
- 記事詳細ページ
- 記事投稿ページ
- 記事編集ページ
- ログインページ
- ユーザー登録ページ
- プロフィール編集ページ
- ユーザーページ
- 検索結果ページ
- タグ別記事一覧ページ

想定する画面 URL は次の通りである。

```text
/                トップページ
/articles         記事一覧
/articles/{id}    記事詳細
/articles/new     記事投稿
/articles/{id}/edit 記事編集
/login            ログイン
/register         ユーザー登録
/users/{id}       ユーザーページ
/search           検索結果
/tags?tag={encodedNormalizedName}   タグ別記事一覧
```

タグ別記事一覧の`tag`には、タグの`normalized_name`を指定する。URLを生成する際は、`URLSearchParams`などを使用して値をURLエンコードする。

React Router を採用し、React 側の画面遷移は React Router で管理する。

共通レイアウト、ナビゲーション設計、画面ごとの詳細レイアウトは後続の画面設計で決定する。

## 10. Phase 1 で実装しないもの

Phase 1 では、次の機能は実装しない。

- 下書き機能
- 非公開記事機能
- slug
- 編集履歴
- 削除済み記事の復元機能
- リアルタイム Markdown プレビュー
- ユーザーごとの非表示閾値設定
- 管理者機能
- アイコン画像
- 画像アップロード
- コメント機能
- ブックマーク機能

画像アップロードに関する API・画面は作成しない。将来画像アップロードを追加する場合は、画像アップロード API および画像管理機能を追加する。将来的には、記事本文に Markdown の画像記法で画像 URL を埋め込む方針とする。

```md
![説明文](/storage/articles/example.png)
```

コメント機能に関する API・画面、ブックマーク機能に関する API・画面は作成しない。
