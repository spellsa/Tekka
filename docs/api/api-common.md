# Tekka API共通仕様書

## 1. 概要

本仕様書では、TekkaのPhase 1で使用するAPIの共通ルールを定義する。

各APIのURL、リクエスト項目、レスポンス項目、バリデーション、権限、処理内容などの詳細は、各API仕様書で定義する。

---

## 2. APIの基本構成

APIのベースURLは、次のとおりとする。

```text
/api
```

各APIは、原則として`/api`以下に定義する。

例:

```text
GET /api/articles
GET /api/articles/{id}
POST /api/articles
```

Phase 1では、次のようなURLによるAPIバージョニングは行わない。

```text
/api/v1/...
```

APIはReact SPAから利用し、LaravelはJSON形式のレスポンスを返す。

---

## 3. 通信形式

APIのリクエストおよびレスポンスには、原則としてJSONを使用する。

```text
Accept: application/json
Content-Type: application/json
```

文字コードはUTF-8とする。

ReactからAPIへリクエストする際は、`Accept: application/json`を共通設定する。

JSON形式のリクエスト本文を送信する場合は、`Content-Type: application/json`を使用する。

リクエスト本文を持たない`GET`および`DELETE`では、`Content-Type`の指定は不要とする。

Phase 1では画像やファイルのアップロードを行わないため、`multipart/form-data`は使用しない。

---

## 4. 認証

認証には、Laravel SanctumのSPA認証を使用する。

認証状態はCookieベースで管理し、CSRF保護を有効にする。

JWTおよびAPIトークンは使用しない。

CSRF Cookieの取得には、Sanctumが提供する次のエンドポイントを使用する。

```text
GET /sanctum/csrf-cookie
```

ログインユーザーの取得には、次のAPIを使用する。

```text
GET /api/user
```

認証が必要なAPIには、各API仕様で認証の要否を明記する。

認証が必要なAPIへ未認証でアクセスした場合は、ログイン画面へのリダイレクトではなく、HTTPステータスコード`401`のJSONレスポンスを返す。

具体的なログイン・ログアウト処理の流れは、認証API仕様書で定義する。

---

## 5. 命名規則

### 5.1 JSONのキー名

JSONのキー名には、`snake_case`を使用する。

例:

```json
{
  "user_id": 1,
  "published_at": "2026-07-19T05:30:00Z"
}
```

### 5.2 API URL

APIのリソース名は、原則として複数形を使用する。

```text
/api/articles
/api/users
/api/tags
```

作成・更新・削除などの基本操作は、URLに動詞を含めず、HTTPメソッドで表現する。

```text
POST   /api/articles
PUT    /api/articles/{id}
DELETE /api/articles/{id}
```

ログインやログアウトなど、通常のCRUDに当てはまらない処理では、分かりやすさを優先して動詞を含むURLを使用できる。

```text
POST /api/login
POST /api/logout
```

`GET /api/user`は、Laravel Sanctumの構成に合わせた例外とする。

---

## 6. HTTPメソッド

HTTPメソッドは、次のように使い分ける。

| 操作 | HTTPメソッド |
|---|---|
| 取得 | `GET` |
| 作成 | `POST` |
| 更新 | `PUT` |
| 削除 | `DELETE` |

Phase 1では、`PATCH`は使用しない。

更新APIでは、編集フォームで扱う値をまとめて送信する。

---

## 7. リクエストパラメータ

### 7.1 パスパラメータ

特定のリソースを識別する値は、パスパラメータとして指定する。

```text
GET /api/articles/{id}
GET /api/users/{id}
```

### 7.2 クエリパラメータ

検索条件、絞り込み条件およびページ番号は、クエリパラメータとして指定する。

```text
GET /api/articles?page=2
GET /api/search?q=laravel&page=1
```

一覧の並び順など、APIごとに異なる条件は、各API仕様書で定義する。

### 7.3 JSONリクエストボディ

データの作成および更新に使用する値は、JSON形式のリクエストボディに格納する。

```json
{
  "title": "記事タイトル",
  "body": "# 本文",
  "tags": ["Laravel", "React"]
}
```

`GET`および`DELETE`では、原則としてリクエストボディを使用しない。

`POST`および`PUT`では、必要に応じてJSON形式のリクエストボディを使用する。

任意の文字列項目へ空文字が送信された場合は、Laravel側で`null`として扱う。

必須項目へ空文字が送信された場合は、バリデーションエラーとする。

---

## 8. 日時形式

APIで扱う日時は、ISO 8601形式の文字列とする。

サーバー側では日時をUTCで保存・処理し、APIはUTCで返す。UTCを表す`Z`を付ける。

```json
{
  "created_at": "2026-07-19T05:30:00Z",
  "updated_at": "2026-07-19T06:00:00Z"
}
```

画面上での日時表示は、React側でUTCから日本時間（JST）へ変換する。

日時が設定されていない場合は、空文字ではなく`null`を返す。

```json
{
  "published_at": null
}
```

---

## 9. 成功レスポンス

APIがデータを返す場合は、レスポンスデータを`data`に格納する。

### 9.1 単一データ

単一のデータを返す場合、`data`はオブジェクトとする。

```json
{
  "data": {
    "id": 1,
    "title": "記事タイトル"
  }
}
```

### 9.2 一覧データ

複数のデータを返す場合、`data`は配列とする。

```json
{
  "data": [
    {
      "id": 1,
      "title": "記事タイトル"
    }
  ]
}
```

処理結果を画面へ通知する必要がある場合は、`message`を追加できる。

すべての成功レスポンスに`message`を付ける必要はない。

削除成功時は、原則として`204 No Content`を返し、レスポンス本文を返さない。

axiosでは、HTTPレスポンス本文が`response.data`に格納されるため、APIレスポンス内の`data`は次のように参照する。

```js
const article = response.data.data;
```

最初の`data`はaxiosが保持するレスポンス本文であり、2つ目の`data`はTekkaのAPIレスポンスで定義したキーである。

---

## 10. エラーレスポンス

APIの処理に失敗した場合は、エラー内容を`message`に格納する。

```json
{
  "message": "指定された記事が見つかりません。"
}
```

バリデーションエラーでも、Phase 1では入力項目ごとの`errors`は返さない。

```json
{
  "message": "入力内容に問題があります。"
}
```

独自のエラーコードは定義しない。

React側でエラーの種類を判定する場合は、`message`の文字列ではなくHTTPステータスコードを使用する。

---

## 11. ページネーション

次の一覧系APIでは、Laravel標準のページ番号方式によるページネーションを使用する。

- 記事一覧
- 検索結果
- タグ別記事一覧
- ユーザーページの記事一覧

ページ番号は、`page`クエリパラメータで指定する。

```text
GET /api/articles?page=2
```

1ページあたりの件数は20件固定とする。

Phase 1では、`per_page`による件数変更には対応しない。

レスポンスには、一覧データを`data`、ページ情報を`meta`として返す。

```json
{
  "data": [
    {
      "id": 21,
      "title": "記事タイトル"
    }
  ],
  "meta": {
    "current_page": 2,
    "last_page": 5,
    "per_page": 20,
    "total": 91
  }
}
```

React側では、最低限「前へ」「次へ」のページ移動UIを用意する。

検索条件や絞り込み条件が変更された場合は、ページ番号を1へ戻す。

---

## 12. 存在しないデータの扱い

### 12.1 単一データ

指定された記事やユーザーなどが存在しない場合は、`404 Not Found`を返す。

```json
{
  "message": "指定された記事が見つかりません。"
}
```

### 12.2 一覧データ

一覧や検索結果が0件の場合は、エラーにしない。

`200 OK`と空配列を返す。

```json
{
  "data": [],
  "meta": {
    "current_page": 1,
    "last_page": 1,
    "per_page": 20,
    "total": 0
  }
}
```

### 12.3 論理削除済みデータ

論理削除済みの記事は、通常のAPIでは存在しないものとして扱い、`404 Not Found`を返す。

削除済みであることを示す専用レスポンスや`410 Gone`は使用しない。

---

## 13. 認可

認証と認可は、次のように区別する。

- 認証: 操作しているユーザーが誰かを確認する
- 認可: そのユーザーが操作を実行できるか確認する

### 13.1 未認証

認証が必要なAPIへ未ログインでアクセスした場合は、`401 Unauthorized`を返す。

```json
{
  "message": "認証が必要です。"
}
```

### 13.2 権限不足

ログイン済みだが操作権限がない場合は、`403 Forbidden`を返す。

対象例:

- 他人の記事の編集
- 他人の記事の削除
- 他人のプロフィール更新
- 自分の記事への評価

```json
{
  "message": "この操作を行う権限がありません。"
}
```

対象自体が存在しない場合は`404`、対象は存在するが権限がない場合は`403`とする。

---

## 14. HTTPステータスコード

Phase 1では、最小限のHTTPステータスコードのみ使用する。

| コード | 用途 |
|---:|---|
| `200` | 取得・更新成功 |
| `201` | 作成成功 |
| `204` | 削除成功 |
| `401` | 未認証 |
| `403` | 権限不足 |
| `404` | 対象が存在しない |
| `422` | 入力値のバリデーションエラー |

各APIが実際に返すステータスコードは、それぞれのAPI仕様書に記載する。

---

## 15. トランザクション

複数のテーブルまたは複数のレコードを更新し、それらを一体として成功させる必要がある処理では、DBトランザクションを使用する。

処理途中で失敗した場合は、トランザクション内の変更をすべて取り消す。

主な対象候補は、次のとおりとする。

- 記事投稿とタグの関連付け
- 記事編集とタグの更新
- 記事削除とユーザースコアの再計算
- 評価の作成・変更・取り消し
- 記事スコアとユーザースコアの更新

単一テーブルへの単純な1回の更新では、原則としてトランザクションを必須としない。

具体的なトランザクション対象と処理手順は、各API仕様書で定義する。

---

## 16. レスポンスに含める情報

### 16.1 基本方針

DBカラムをそのまますべて返さない。

API Resourceなどを使用し、画面表示や処理に必要な返却項目を明示する。

### 16.2 非公開情報

次の情報は、APIレスポンスに含めない。

- パスワード
- メールアドレス

Phase 1では、ログインユーザー取得API`GET /api/user`でもメールアドレスを返さない。

### 16.3 関連データ

画面表示に必要な最小限の関連データは、レスポンス内へ埋め込む。

記事レスポンスの例:

```json
{
  "data": {
    "id": 1,
    "title": "記事タイトル",
    "author": {
      "id": 3,
      "username": "User"
    },
    "tags": [
      {
        "id": 1,
        "display_name": "Laravel"
      }
    ]
  }
}
```

記事レスポンスに投稿者IDだけを返し、投稿者取得APIを別途呼び出す設計にはしない。

ただし、関連情報を無制限に含めることは避ける。

次のような情報は含めない。

- 投稿者のメールアドレス
- 投稿者の全記事
- 記事に対する全評価レコード
- DB内部でのみ使用する値

各APIで具体的に返す項目は、それぞれのAPI仕様書で定義する。
