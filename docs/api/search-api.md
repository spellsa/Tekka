# Tekka 検索API仕様書

## 1. 概要

本仕様書では、TekkaのPhase 1で使用する検索APIを定義する。

検索APIでは、記事タイトルまたは記事本文に指定キーワードを含む記事を取得する。

API全体の共通ルールについては、`api-common.md`を参照する。

---

## 2. 検索API一覧

| メソッド | URL           | 認証 | 概要                               |
| -------- | ------------- | ---- | ---------------------------------- |
| `GET`    | `/api/search` | 不要 | キーワードを使用して記事を検索する |

使用例:

```text
GET /api/search?q=laravel&page=1
```

---

# 3. 記事検索

## 3.1 基本情報

| 項目     | 内容                                                       |
| -------- | ---------------------------------------------------------- |
| メソッド | `GET`                                                      |
| URL      | `/api/search`                                              |
| 認証     | 不要                                                       |
| 概要     | 記事タイトルまたは記事本文にキーワードを含む記事を取得する |

## 3.2 クエリパラメータ

| 項目   | 型      | 必須 | 内容                          |
| ------ | ------- | ---: | ----------------------------- |
| `q`    | string  | 必須 | 検索キーワード                |
| `page` | integer | 不要 | ページ番号。省略時は1ページ目 |

ページネーションにはLaravel標準のページ番号方式を使用する。

1ページあたりの取得件数は20件固定とし、`per_page`による変更には対応しない。

```text
GET /api/search?q=sanctum&page=2
```

---

# 4. 検索キーワードの処理

## 4.1 正規化

`q`に対して、検索前に次の処理を行う。

```text
1. URLデコード済みの値を取得する
2. 前後の空白を除去する
3. Unicode NFKC正規化を行う
```

例:

```text
"  Ｌａｒａｖｅｌ  "
↓
"Laravel"
```

タグ名の正規化とは異なり、API側で明示的な小文字化は行わない。

英字の大文字と小文字を区別するかどうかは、MySQLのカラム照合順序に従う。

大文字と小文字を区別しない照合順序を使用する場合、次の検索は同じ結果になる。

```text
Laravel
laravel
LARAVEL
```

## 4.2 文字数

検索キーワードは、正規化後に次の条件を満たす必要がある。

| 条件       | 内容    |
| ---------- | ------- |
| 最小文字数 | 1文字   |
| 最大文字数 | 100文字 |

正規化後に100文字を超える場合は、`422 Unprocessable Content`を返す。

## 4.3 空の検索キーワード

次のリクエストは、バリデーションエラーとする。

```text
GET /api/search
GET /api/search?q=
GET /api/search?q=%20%20%20
```

空の検索を記事一覧として扱うと、`GET /api/articles`と役割が重複するためである。

通常の記事一覧を取得する場合は、次のAPIを使用する。

```text
GET /api/articles
```

---

# 5. 検索対象

検索対象は次の2項目とする。

- 記事タイトル
- 記事本文

条件は次のとおりである。

```text
titleに検索キーワードを含む
または
bodyに検索キーワードを含む
```

概念的なSQL条件は次のとおりである。

```sql
WHERE title LIKE '%検索キーワード%'
   OR body LIKE '%検索キーワード%'
```

検索キーワードをSQLへ直接連結せず、Laravelのクエリビルダによるバインドを使用する。

---

# 6. 検索方式

## 6.1 部分一致

Phase 1では、単純な部分一致検索を使用する。

例えば、検索キーワードが次の場合とする。

```text
Sanctum
```

次のような記事が検索対象になる。

```text
タイトル:
Laravel Sanctumの基本

本文:
この記事ではSanctumを使用して認証を実装します。
```

## 6.2 空白を含む検索キーワード

空白を含む検索キーワードは、1つの連続した文字列として検索する。

```text
Laravel Sanctum
```

この場合、タイトルまたは本文に`Laravel Sanctum`という並びが含まれる記事を取得する。

次のような複数単語のAND検索にはしない。

```text
Laravelを含む
かつ
Sanctumを含む
```

検索キーワードの分割、AND検索、OR検索はPhase 1では実装しない。

## 6.3 SQLワイルドカード

`LIKE`検索では、次の文字がワイルドカードとして扱われる。

```text
%
_
```

利用者が入力した`%`や`_`は、検索用ワイルドカードではなく通常の文字として扱う。

そのため、Laravel側で必要なエスケープを行う。

```text
入力した%
→ 文字としての%を検索する

入力した_
→ 文字としての_を検索する
```

---

# 7. 検索対象に含めない項目

Phase 1では、次の項目はキーワード検索の対象にしない。

- タグ名
- 投稿者名
- ユーザープロフィール
- 記事スコア
- ユーザースコア
- 投稿日時
- 更新日時

タグから記事を探す場合は、タグ別記事一覧APIを使用する。

```text
GET /api/tag-articles?tag=laravel
```

---

# 8. 記事の表示条件

検索結果には、次のすべてを満たす記事だけを含める。

```text
検索キーワードがタイトルまたは本文に含まれる
論理削除されていない
記事スコアが-2以上
投稿者のユーザースコアが91以上
```

次の記事は検索結果から除外する。

- 論理削除済みの記事
- 記事スコアが`-3`以下の記事
- 投稿者のユーザースコアが`90`以下の記事

この条件は、通常の記事一覧およびタグ別記事一覧と同じとする。

ユーザー別記事一覧とは異なり、記事スコアおよびユーザースコアによる非表示を適用する。

---

# 9. 並び順

検索結果は、記事の作成日時が新しい順とする。

```text
created_at DESC
```

検索キーワードとの関連度順には並べない。

Phase 1では、次の並び替えには対応しない。

- 関連度順
- 記事スコア順
- 更新日時順
- 古い順

---

# 10. レスポンス形式

記事データには、`articles-api.md`で定義した記事一覧用の共通形式を使用する。

| 項目           | 型      | 内容                     |
| -------------- | ------- | ------------------------ |
| `id`           | integer | 記事ID                   |
| `title`        | string  | 記事タイトル             |
| `excerpt`      | string  | 記事本文から生成した概要 |
| `score`        | integer | 記事スコア               |
| `published_at` | string  | 投稿日時                 |
| `author`       | object  | 投稿者の概要情報         |
| `tags`         | array   | 記事に付けられたタグ     |

検索結果には、次の項目を含めない。

- 記事本文全体の`body`
- `current_user_evaluation`
- 検索キーワードの一致位置
- 強調表示用のHTML

## 10.1 `excerpt`の扱い

検索結果でも、通常の記事一覧と同じ`excerpt`を使用する。

検索キーワードが本文の後半に存在する場合でも、検索箇所周辺を抜き出す処理は行わない。

```text
bodyの先頭200文字
→ excerpt
```

検索キーワードが本文の1,000文字目にある場合、`excerpt`に検索キーワードが表示されない可能性がある。

検索箇所周辺を表示するスニペット生成は、Phase 1では実装しない。

# 12. 検索結果が0件の場合

検索条件に一致する記事がない場合は、`404`ではなく`200 OK`と空配列を返す。

検索キーワード自体は有効であるため、結果が0件でもエラーにはしない。

<details>
<summary>サンプルレスポンス</summary>

```http
HTTP/1.1 200 OK
Content-Type: application/json
```

```json
{
  "data": [],
  "meta": {
    "current_page": 1,
    "last_page": 1,
    "per_page": 20,
    "total": 0
  }
}
```

</details>

---

# 13. 最終ページを超える場合

最終ページを超えるページ番号を指定した場合は、`200 OK`と空配列を返す。

`meta`には実際のページネーション情報を返す。

<details>
<summary>サンプルレスポンス</summary>

```http
HTTP/1.1 200 OK
Content-Type: application/json
```

```json
{
  "data": [],
  "meta": {
    "current_page": 3,
    "last_page": 1,
    "per_page": 20,
    "total": 5
  }
}
```

</details>

# 14. バリデーションエラー

## 14.1 エラー条件

次の場合は、`422 Unprocessable Content`を返す。

- `q`が指定されていない
- `q`が文字列ではない
- `q`が正規化後に空文字となる
- `q`が正規化後に100文字を超える
- `page`が正の整数ではない

## 14.2 サンプル

<details>
<summary>検索キーワード未指定時のサンプルリクエスト</summary>

```http
GET /api/search?page=1
Accept: application/json
```

</details>

<details>
<summary>サンプルレスポンス</summary>

```http
HTTP/1.1 422 Unprocessable Content
Content-Type: application/json
```

```json
{
  "message": "入力内容に問題があります。"
}
```

</details>

# 15. 処理内容

```text
1. qとpageを検証する
2. qの前後の空白を除去する
3. qをNFKC正規化する
4. LIKE検索用の特殊文字をエスケープする
5. タイトルまたは本文にqを含む記事を検索する
6. 論理削除済みの記事を除外する
7. 記事スコアおよびユーザースコアによる非表示条件を適用する
8. created_atの降順で並べる
9. 20件単位でページネーションする
10. 記事一覧とページネーション情報を返す
```

このAPIはデータを更新しないため、DBトランザクションは使用しない。

---

# 16. DBインデックス

Phase 1では、キーワード検索専用のインデックスおよび全文検索インデックスは作成しない。

記事タイトルと記事本文に対して、単純な`LIKE`による部分一致検索を行う。

期限内の最小実装を優先し、検索性能の改善は後のフェーズで検討する。

---

# 17. ステータスコード一覧

| 状況                   | ステータス |
| ---------------------- | ---------: |
| 検索成功               |      `200` |
| 検索結果が0件          |      `200` |
| 最終ページを超えている |      `200` |
| `q`または`page`が不正  |      `422` |

認証不要の取得APIであるため、通常は`401`、`403`、`404`を使用しない。
